# Backlog

Suggestions from a close read of the code, the methodology document, and the git history. Ordered roughly by leverage, not urgency.

## Documentation drift

- **`Evacuation_Simulator_Methodology.docx` is v1.0 and already behind the code.** It documents `MC_RUNS = 100`; the code runs 500. It lists "Counterfactual MC" for information value as a §13 *future extension*; `computeInfoValue`/`assignForInfoValue` already implement exactly that. It doesn't mention the greedy urgency-ordered assignment with capacity contention (§7 in-app methodology), describing independent assignment instead. It predates the Willingness → gatekeeper change made in this session. Recommend bumping it to v1.1 with a short changelog section, or adding a `CHANGELOG.md` that the docx references rather than trying to keep prose in sync with code on every change.
- **The docx claims to be "auto-generated from the simulator source code structure"** (closing line). It isn't — it's manually authored prose that happens to track the code. Either build a real generation step (even a script that extracts `PARAMS`/`PARAM_DOCS` into a table) or soften the claim, since an examiner who checks will find hand-written text, not generated output.
- **`explainer.html` had drifted from `index.html` in at least three places before this session** (MC run count, "two gatekeepers" vs the code's structure, and an assignment-independence claim that was no longer true). It's now corrected and cross-linked via `#card-N` anchors, but nothing prevents it from drifting again on the next feature change. Since it's a second prose description of the same model as the in-app Methodology accordion, consider whether it should exist as *generated* HTML from a shared source (e.g. a JSON/YAML description of factors, controls, and cards) rather than hand-maintained twice.

## Modelling

- **Willingness's promotion to gatekeeper status is a substantive behavior change, not just a label fix.** In a default-seed run (seed 42, N=8, M=3) it took the excluded-destination count from roughly 2/8 to 5/8, because gatekeeper factors are generated with a 24% blocked-probability weight vs. 10% for standard factors. That's the intended fix (an unwilling host should exclude a site), but it changes the "shape" of every scenario materially. Before citing results from this version in the thesis, re-run the seed sweep / sensitivity analysis the methodology doc already recommends in §11, specifically on `GATEKEEPER_WEIGHT` and the gatekeeper status-distribution array (`[.30,.28,.24,.18]`), to confirm the new exclusion rate is intentional and not an artifact of reusing the security/authority distribution for a conceptually different factor.
- **§13 of the methodology doc lists five other future extensions** (factor correlation, temporal dynamics, real data integration, a counterfactual "what if" UI) that remain undone. Of these, **factor correlation** (security and consent plausibly co-vary in real conflict) is probably the highest-value next step for ecological validity, since it's flagged as a known limitation in both the doc and the in-app §7 Assumptions section.
- **The 40% success threshold and the ±1 ordinal perturbation are both acknowledged-uncalibrated assumptions.** If empirical UNHCR/IOM displacement data (the "Real data integration" extension) ever gets pulled in, these are the two parameters most worth checking against it first.

## Repo hygiene

- **Two GitHub remotes exist for this working copy** (`colab` → `Ethical-Tech-CoLab/India-EvacSimulation`, `origin` → `indiaclarke03-ops/EvacSimulator`), and until this session `EvacSimulator` held an older, simpler snapshot of the tool (no greedy assignment, no alerts, no info-value panel, no gatekeeper-labelled exclusions). Decide which repo is canonical going forward and say so in both READMEs (this session's docs do this), otherwise the two will silently re-diverge the same way `index.html`/`evacuation-simulator.html` did within this repo.
- **`index.html` and `evacuation-simulator.html` existed as two separately-committed copies of the same file**, with the most recent feature commit ("7 simulation improvements") landing only in `evacuation-simulator.html` — meaning GitHub Pages (which serves `index.html`) was showing a stale build. Fixed in this session by making `index.html` canonical and turning `evacuation-simulator.html` into a redirect stub; worth a pre-commit check or CI step going forward to catch this class of drift automatically (e.g. fail if two HTML files in root have >90% textual similarity).
- **A 15MB `ACLED Data_2026-06-01.csv` and several UNHCR "5W" `.xlsx` files sit in the repo root**, unreferenced by the application (confirmed via grep — no `.csv`/`.xlsx` path appears in `index.html`). They inflate clone size for a tool that is explicitly synthetic-data-only. Recommend moving them to a `research-data/` subfolder with a short note on provenance, or removing them from git history entirely with `git filter-repo` if they're not needed for the thesis appendix.
- **No `LICENSE` file** in either repo. Worth adding even a permissive default if this is meant to be shared with examiners or the public.
- **No automated tests.** `computeReadiness`, `perturbStatus`, `monteCarlo`, `compositeScore`, and `assign` are all pure(ish) functions with no DOM dependency — good unit-test candidates. Even a handful of tests pinning known seed → known output would catch accidental regressions in the scoring formulas, which is exactly the kind of silent-drift risk this backlog keeps surfacing elsewhere.

## Smaller UI notes
